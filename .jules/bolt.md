## 2025-03-08 - Prevent Global Re-renders on Form Input
**Learning:** Using `const { a, b } = useCreateStore()` destructured directly without a selector in Zustand causes the component to re-render whenever *any* state in the store changes. Because `useCreateStore` holds rapid-changing form state like `userInput` (updated on every keystroke), this caused all components connected to the store (e.g., `CreateFlow`, `ResultEditor`, etc.) to re-render constantly.
**Action:** Always use granular selectors `const a = useCreateStore(s => s.a)` or `useShallow` from `zustand/react/shallow` when accessing store properties to restrict re-renders to only when the specifically requested fields change.

## 2024-03-24 - [Parallelize DB Queries with Promise.all in stats]
**Learning:** Found an opportunity to replace 7 sequential independent database queries with \`Promise.all()\` in the \`GET /api/stats\` endpoint. The queries fetched independent statistics directly from D1. A specific nuance here is that the \`scheduling\` query may fail on unmigrated local environments since it references a \`scheduled_at\` column. To include it in \`Promise.all\` without causing the entire batch to fail, we need to handle the exception gracefully at the individual promise level by attaching \`.catch(() => null)\`. This maintains the original fallback behavior while unlocking concurrency for the rest of the queries. Replacing sequential \`await db.prepare().first()/all()\` with \`Promise.all()\` in Cloudflare D1 reduces overall API response latency significantly, as it only blocks on the slowest query rather than the sum of all their execution times.
**Action:** Always identify independent database queries in Hono/Cloudflare Worker route handlers and group them using \`Promise.all()\`. Specifically, handle schema-migration differences or optional queries with local \`.catch()\` blocks attached to their individual promises instead of a massive outer try-catch.
